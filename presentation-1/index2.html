Pseudo Elements

Pseudo elements
Just like pseudo-classes, pseudo-elements are added to selectors but instead of describing a special state, they allow you to style certain parts of a document. For example, the ::first-line pseudo-element targets only the first line of an element specified by the selector.(MDN)
Pseudo elements
When their computed content value is not none, these pseudo-elements generate boxes as if they were immediate children of their originating element, and can be styled exactly like any normal document-sourced element in the document tree. They inherit any inheritable properties from their originating element; non-inheritable properties take their initial values as usual. (w3c spec)
Pseudo elements - syntax
selector::pseudo-element { property: value; }

Generated Content - :before and :after
  • The CSS ::before and ::after pseudo-elements match a virtual first/last child of the selected element.
  • They are typically used to add cosmetic content to an element by using the content CSS property.
  • These pseudo elements' display property have an initial value of inline by default.
Generated content - Accessibility issues

Genrated content is not part of the DOM, this means:

  • The content will not be accessible to (some?) screen readers
  • The content will not be selectable (so no copy/paste)
  • Bots won't see the content for SEO purposes
  • Can't be set by javascript. Although you can retrieve the contents with window.getComputedStyle
see CSS generated content is not content
The content property
The content CSS property is used with the ::before and ::after pseudo-elements to generate content in an element. Objects inserted using the content property are anonymous replaced elements.

The attr() CSS function
  • The attr() CSS function is used to retrieve the value of an attribute of the selected element and use it in the style sheet. It can be used on pseudo-elements too and, in this case, the value of the attribute on the pseudo-element's originated element is returned.
  • The attr() function can be used with any CSS property, but support for properties other than content is experimental.

Specificity

Possibly the worst part of CSS.
#1. Do We Actually Need Specificity In CSS? (Philip Walton)

For anyone who’s a little rusty on how the cascade works, for normal CSS rules, the cascade takes three things into consideration: source order, specificity, and importance.

Source order makes a lot of sense. It’s natural and intuitive to think that if rule Y comes after rule X, and both rules apply to the same element, rule Y’s declarations should “win”.

#2. Do We Actually Need Specificity In CSS? (Philip Walton)

Importance also makes a lot of sense. There are always going to be cases where you need to override something, and it’s good to have the option to do so. Importance is also the only way to override inline styles, so it’s actually quite necessary in some cases.

#3. Do We Actually Need Specificity In CSS? (Philip Walton)

But I think specificity is different.

Specificity isn’t intuitive, and—especially for new developers—the results can often seem like a gotcha rather than the intended behavior. I’m also not sure there’s an equivalent in other systems or languages.

#4. Do We Actually Need Specificity In CSS? (Philip Walton)

For example, what if specificity were a thing in JavaScript. Imagine how much more unpredictable your code would be if the following test passed.

window.document.foo = 'bar';
document.foo = 'qux';

assert(window.document.foo == 'bar'); //true !!?!
#5. Do We Actually Need Specificity In CSS? (Philip Walton)

It would be crazy and completely unmanageable if the window.document.foo = 'bar' assignment above (which comes first) trumped the document.foo = 'qux' assignment (which comes second) just because the reference was “more specific”. Yet that’s essentially what happens in CSS.

Calculating CSS Specificity Value

  • If the element has inline styling, that automatically wins (1,0,0,0 points)
  • For each ID value, apply 0,1,0,0 points
  • For each class value (or pseudo-class or attribute selector), apply 0,0,1,0 points
  • For each type/pseudo element reference, apply 0,0,0,1 point

The Cascade

The cascade is an algorithm which takes a unordered list of declared values for a given property on a given element, sorts them by their declaration’s precedence, and outputs a single cascaded value.

The Cascade Algorithm

The cascade sorts declarations according to the following criteria, in descending order of priority: (intentionally simplified:)

  • Origin and Importance
  • Specificity
  • Order of Appearance
The Cascade
  • The output of the cascade is a (potentially empty) sorted list of declared values for each property on each element.
  • When the cascade does not result in a value, the specified value must be found some other way. Inherited properties draw their defaults from their parent element through inheritance; all other properties take their initial value. Authors can explicitly request inheritance or initialization via the inherit and initial keywords.

Quantity Queries

...make CSS quantity-aware using specially formed selectors.
Question:

Write a CSS rule to style an element with a blue background only if it's parent has one child.
If it's parent has more than one child - give the children a green background

Answer:

* {
  background-color: green;
}
:first-child:last-child {
  background-color: blue;
}

OR

The truth is that CSS has a built-in selector for this :only-child


* {
  background-color: green;
}
:only-child {
  background-color: blue;
}
Taking this idea further for n elements:

Style the children when the parent has At least 5 elements


:nth-last-child(n+5), :nth-last-child(n+5) ~ * {
  /* styles go here */
}

Style the children when the parent has At most 5 elements


:nth-last-child(-n+5):first-child, :nth-last-child(-n+5):first-child ~ * {
  /* styles go here */
}

Selectors - General notes

  • Type and attribute names aren't case sensitive. Attribute values (like classes) are case sensitive
  • Selectors traverse the DOM downward, so no parent or previous sibling selector (for now)
  • Think of pseudo classes like filters on the matched element which limit the match to a given state
  • Pseudo classes may be chained to add additional filters (order doesn't matter)
  • Selector syntax :pseudo-class::pseudo-element eg. .myClass:hover:before { ... } (order matters!!)

Selectors - General notes

  • Selectors are parsed from right to left for efficiency reasons.
    (See StackOverflow: Why do browsers match CSS selectors from right to left? ;
    CSS combinator precedence? )

    If you were to read selectors in plain English, you would read them from right to left too. So, the selector:

    a > b ~ c d

    would mean:

    Select any d element
    that is a descendant of a c element
    that is a sibling of, and comes after, a b element
    that is a direct child of an a element.

    • Best practices: (cssguidelin.es/#css-selectors)

    • Select what you want explicitly, rather than relying on circumstance or coincidence.
    • Write selectors for reusability, so that you can work more efficiently and reduce waste and repetition.
    • Do not nest selectors unnecessarily, because this will increase specificity and affect where else you can use your styles.
    • Do not qualify selectors unnecessarily, as this will impact the number of different elements you can apply styles to.
    • Keep selectors as short as possible, in order to keep specificity down and performance up.

    1/100th of the maximum value between the height and the width of the viewport.
